# Skill: Observability (Logs, Metrics, Traces)

**Description**: Паттерны сбора информации о работе микросервиса.

## Три кита Observability

### 1. Structured Logging (Логи)
- Логировать события исключительно в структурированном виде (JSON).
- Ключевые поля: `timestamp`, `level`, `message`, `service_name`, `trace_id`, `actor_id` (если есть).
- Не использовать уровни логов произвольно. `ERROR` - требует внимания, `WARN` - предупреждение, `INFO` - бизнес-процесс, `DEBUG` - только для локальной отладки.
- **NEVER** логировать: пароли, PII данные (Email, Имя) без обфускации или хэширования, секретные токены.

### 2. Distributed Tracing (Трейсинг)
- Использовать **OpenTelemetry**.
- По контексту (HTTP headers или metadata) передавать `Trace ID` (уникальный идентификатор всего пайплайна) и `Span ID` (шаг этого пайплайна).
- Трейс проходит через: API Gateway -> Service A -> Database -> Message Queue -> Service B. Без `trace_id` нереально свести логи разных микросервисов.
- Заголовки: стандарты W3C Trace Context (`traceparent`, `tracestate`).

### 3. Metrics (Метрики - Prometheus)
- Инструменты: скрейпинг `Prometheus`, визуализация `Grafana`.
- Следовать методологии **RED** для каждого сервиса/эндпоинта:
  - **R (Rate)**: Количество запросов в секунду (RPS).
  - **N (Network/Node)**: Сетевые ошибки или потеря связи с нодами БД.
  - **E (Errors)**: Количество ошибок в секунду (или доля 5xx статусов).
  - **D (Duration)**: Время выполнения запросов (Latency, Percentiles 95th, 99th - а не среднее).
- Следовать методологии **USE** для инфраструктуры/worker'ов:
  - **U (Utilization)**: Насколько ресурс занят (например, загрузка CPU, заполненность пула коннектов к БД).
  - **S (Saturation)**: Очередь (сколько сообщений ждут обработки).
  - **E (Errors)**: Аппаратные или системные ошибки.

## Alerting
- Алерты должны настраиваться на SLA/SLO, а не на технические детали. "Загрузка CPU 90%" — это не повод будить дежурного, если пользователи этого не замечают. "Доля успешных checkout < 99.5%" — повод будить дежурного немедленно.

## Контекст Выполнения (Inputs)
- Если вы отлаживаете проблему (`<issue-description>`), используйте эти метрики и логи для локализации Root Cause.
- Если разрабатываете новую фичу (`<feature-name>` или `<endpoint-name>`), убедитесь, что в код добавлены структурированные логи и RED-метрики конкретно для этого элемента.
